# cauldron-psi — project-wide semantic index (design)

*Design of record for the PSI-style semantic layer. Produced 2026-07-12 by a research panel over
IntelliJ Platform PSI/indexing internals, intellij-rust stubs/resolve2, and C call-graph reality
(cFS idioms), then synthesized. Companion to docs/phase0.md (NASA lint Areas 3/5) and scope.md P7.*

**Why this exists.** JetBrains' decisive advantage is not the editor — it is the PSI: a full,
incrementally-updated, indexed semantic model of the whole project. LSP hands back opaque per-file
diagnostics; a project-wide queryable index can express WHOLE-PROGRAM properties. JPL Power-of-Ten
Rule 1 (no recursion) is fundamentally a call-graph property — per-TU tools cannot see a cross-file
cycle. Cauldron's job is "catch my errors before NASA CI does"; this index is the substrate for
both the NASA whole-program checks and, later, find-usages/rename refactoring.


## Architecture

NEW CRATE crates/cauldron-psi in the cauldron repo (4th family member; headless lib + tiny [[bin]] psi-spike; deps: tree-sitter 0.22 + tree-sitter-c 0.21 (the SAME workspace pin as cauldron-editor/-lint — never bump alone), rayon, ignore, smol_str, parking_lot; NO egui/eframe, mirroring the cauldron-lint boundary rule: independently useful + must run headless in the cFS CI loop).

MODULE TREE: collect.rs (pure per-file extraction), index.rs (interners + forward/inverted maps), graph.rs (resolution + Tarjan + witnesses), invalidate.rs (generation counter + changed-set + two-hash check), query.rs (public API). persist.rs deliberately ABSENT in v1 (see storage).

CORE TYPES:
- FileId(u32), Sym(u32) — path and name interners (affordable at cFS ~5k-file scale; interning removes IdIndex-style hash-collision rechecks for the def index).
- SymKey — the linkage-aware symbol identity, the single most load-bearing type: enum { External(Sym), Internal(FileId, Sym) }. C has no overloading, so name+linkage resolution is EXACT for direct calls; this is what makes the whole design work and what a name-only key would break (phantom SCCs from duplicate statics).
- Stub { name: Sym, kind: FnDef|FnDecl|MacroDef|Typedef|Tag|EnumConst|GlobalVar|Include, flags: u32 bitfield (is_static, is_inline, is_variadic, is_function_like_macro, has_token_paste, ...), byte_range, name_range, body_lines, arity: Option<u8> }. Stubs are PLAIN DATA — name+kind+span, never node handles. Navigation from a stub re-opens the file and finds the node at the span (re-anchor by name if drifted). This deliberately skips IntelliJ's stub<->AST dual-backed PsiElement, their single largest complexity/crash source.
- FileFacts { path, interface_hash: u64, body_hash: u64, stubs: Vec<Stub>, calls: Vec<CallSite{caller_stub: u32, callee: Sym, offset, kind: Direct|MacroMined}>, address_taken: Vec<(Sym, offset, ctx: Initializer|AddrOf|Assign|Argument)>, idents: Vec<(Sym, ContextMask code|comment|string)>, includes: Vec<SmolStr>, error_bytes: u32 } — the ONE per-file artifact, produced by ONE tree-sitter walk. Pure function of file text: trivially rayon-parallel, property-testable, and stub list + all index contributions can never disagree (the StubUpdatingIndex atomicity lesson).
- Index { files: HashMap<FileId, Arc<FileFacts>> (simultaneously forward index and per-file cache), defs_by_name: HashMap<Sym, SmallVec<(FileId, stub_idx)>>, decls_by_name, files_with_ident: HashMap<Sym, (SmallVec<FileId>, ContextMask)> (the IdIndex analog, phase-1 of find-usages), callers_of: HashMap<Sym, Vec<(FileId, CallSite)>> (materialized — the one thing JetBrains doesn't materialize, but whole-graph R1 queries + cFS scale make it cheap and right), address_taken_set, include_map, missed_includes (unresolved #include paths -> precise invalidation on file creation), generation: u64, status: Ready|Indexing{done,total} }.
- CallGraph — a DERIVED, memoized artifact stamped with the generation it was built at: linkage-keyed nodes (functions + function-like macros as first-class nodes + external leaves for decl-without-def), Tier-1 edges (direct + macro-mined, name/linkage-resolved), Tier-2 edges (indirect site -> arity-filtered address-taken set), SCC condensation + witness cycles. NEVER incrementally maintained — full Tarjan recompute on demand is single-digit ms at cFS scale; incremental SCC is a research problem we refuse to have.

STORAGE: pure in-memory, plain HashMaps. No sled, no persistence in v1 — this mirrors resolve2 itself (CrateDefMap is never persisted; only stubs are) taken one step further: at 5k files a full cold rayon index is seconds, so we persist NOTHING and rebuild on startup. A per-file bincode blob cache keyed (path, size, mtime, psi_version = tree-sitter grammar version + stub schema version) is the designed-but-unbuilt escape hatch, added only if cold start ever annoys. This deletes an entire class of work (transactional on-disk index engine, version migration) that IntelliJ needs for 100k-file monorepos and we don't.

THREADING: PsiService owns the Index behind parking_lot::RwLock inside the cauldron app; one worker thread does all writes (initial scan fans FileFacts extraction out over rayon, merges single-threaded; incremental updates + graph recomputes run on the same worker). Queries clone an Arc snapshot under a brief read lock and run lock-free; every query result carries the generation it was computed at, and consumers drop stale results by generation compare (the codebase's existing stamped-stale-drop pattern from the LSP design). Results reach the UI via std::sync::mpsc + ctx.request_repaint() — the exact cider PTY template every worker in this codebase already uses. Nothing PSI-related ever runs on the frame thread; the 8 ms editor budget is untouched.


## Indexing pipeline

INITIAL SCAN (cold start, background worker): walk workspace root with the ignore crate (gitignore-aware — same walker the tree/quick-open/search already standardize on), filter *.c/*.h, apply partition globs from .cauldron/nasa.toml (defaults exclude **/ut-stubs/**, **/unit-test*/**, **/coveragetest/** and build dirs — exact cFS directory names verified in the spike; without this, cFS's deliberate same-named stub functions contaminate the graph). rayon par_iter: parse each file with tree-sitter-c, run collect::file_facts(text) -> FileFacts. Extraction is ONE explicit-stack TreeCursor walk (NOT recursive — a Power-of-Ten linter must not itself recurse; also replaces cauldron-lint's current recursive walk()): top-level stubs via the declarator-descent already proven in lib.rs function_name(); descend into function bodies capturing ONLY call_expressions + address-taken identifiers (body detail beyond that is skipped — the RsFunctionStub body-skip lesson, adapted because C bodies DO matter to R1 but only their call sites); macro replacement lists are opaque preproc_arg tokens in tree-sitter-c, so each macro body is RE-PARSED as a C fragment and mined for calls/address-taken names (one level, name-linked — macro-calls-macro chains resolve transitively through the graph, no textual expansion); all #if branches indexed (the sound union over configurations); ERROR-node TOLERANT — extract whatever parsed, never bail on a file, record error_bytes as index-health telemetry. Merge phase: single-threaded fold of FileFacts into the inverted maps; status = Indexing{done,total} throughout, flipping to Ready atomically.

INCREMENTAL UPDATE — one chokepoint, two sources, both funnel into replace_file_facts(FileId, Arc<FileFacts>):
(a) Open buffers: Buffer::apply(Transaction) — already the single mutation chokepoint per phase0 Area 1 — bumps the global generation counter and adds the FileId to changed_files. NOTHING is computed at keystroke time (the resolve2 pull-based rule). After the existing ~200 ms debounce, the worker re-collects FileFacts from the buffer's ALREADY-LIVE incremental tree-sitter tree (a cheap walk, no reparse) and installs it as an OVERLAY shadowing the on-disk facts for that FileId; overlay dropped on save when disk and buffer converge. Buffer-derived facts never touch any persistent path.
(b) Disk: notify watcher (owned by the app, not the psi crate — psi stays passive/pure) marks unopened files dirty on write; create/delete/rename set should_rebuild-class structural flags; file creation consults missed_includes for precise invalidation.

THE TWO-HASH INVALIDATION (resolve2's crown jewel, split for C because bodies carry call sites): interface_hash = hash of the sorted exported surface (defs, decls, macros, typedefs, tags, globals, includes — everything visible to OTHER files); body_hash = hash of the sorted call-site + address-taken list. On drain: recollect facts, compare hashes. Neither changed (typing inside a body without adding/removing a call — the overwhelmingly common edit) -> drop from changed_files, ZERO index work, generation memo of the CallGraph stays valid. body_hash only -> retract/insert this file's callers_of rows + drop the CallGraph memo. interface_hash -> full retract/insert of this file's contributions across all maps + drop memo. Retraction is always via the forward index (the old Arc<FileFacts> lists exactly the keys the file contributed) — diff old vs new keys, never scan an inverted map.

PULL-BASED REBUILD: queries call ensure_up_to_date() first; O(1) fast path is a generation-stamp compare (DefMapService hasLatestStamp pattern); on miss, drain changed_files through the hash check under one build lock, re-checking the fast path after acquiring it. The CallGraph is then lazily recomputed (full Tarjan) only if a query actually needs it and its memo generation is stale.


## Query API

Public surface (crates/cauldron-psi/src/query.rs):

  pub struct Psi;                             // app-owned service handle
  impl Psi { pub fn snapshot(&self) -> Result<PsiSnapshot, NotReady>; pub fn status(&self) -> IndexStatus; }
  // IndexStatus = Ready { generation } | Indexing { done, total }
  impl PsiSnapshot {                          // immutable, generation-stamped, lock-free
    pub fn find_definitions(&self, name: &str) -> Vec<SymbolHit>;   // SymbolHit { key: SymKey, file, byte_range, name_range, kind, flags }
    pub fn find_declarations(&self, name: &str) -> Vec<SymbolHit>;
    pub fn find_usages(&self, name: &str, mask: ContextMask) -> UsageStream;  // two-phase, see below
    pub fn callers(&self, key: &SymKey) -> Vec<CallSiteHit>;        // from materialized callers_of
    pub fn callees(&self, key: &SymKey) -> Vec<CallSiteHit>;
    pub fn call_graph(&self) -> Arc<CallGraph>;                     // memoized vs generation; ms-scale rebuild on miss
    pub fn could_recurse(&self, key: &SymKey) -> Recursion;         // Acyclic | Confirmed(WitnessCycle) | PossibleViaIndirect { cycle, indirect_edges: Vec<IndirectEdge> }
    pub fn recursion_findings(&self) -> RecursionReport;            // whole-program: all Tier-1 SCCs + Tier-2-only SCCs + unresolved/indirect site inventory
    pub fn call_paths(&self, from: &SymKey, to: &SymKey, limit: usize) -> Vec<Vec<SymKey>>;  // BFS over SCC condensation
    pub fn call_depth(&self, root: &SymKey) -> Depth;               // Finite(n) | Unbounded { via_scc } — longest path in condensation DAG; future stack-bound check
    pub fn address_taken(&self, key: &SymKey) -> Vec<Site>;         // feeds R9 and Tier-2 explanations
    pub fn health(&self) -> IndexHealth;                            // per-file ERROR density, unresolved counts
  }

TWO-PHASE FIND-USAGES (the IdIndex/PsiSearchHelper crown jewel, transferred intact): phase 1 = files_with_ident[name] filtered by ContextMask -> small candidate file set, returned instantly; phase 2 = per candidate file, run a tree-sitter query for matching identifiers, then the C resolve check (shadowed by same-named local/param? static def in a different TU? macro-body hit -> flagged "textual — verify"). For open buffers phase 2 is free (live tree). UsageStream yields per-file so the UI paints progressively; the fold checks the generation token each file so a keystroke aborts an in-flight query instead of racing it. Rename (later) = the same machinery with strict resolve, emitting one multi-file batch of Transactions applied end-to-start per file — which is why psi returns exact identifier byte spans everywhere.

NASA LINTER CONSUMPTION: cauldron-lint gains a dependency on cauldron-psi and a new pot/rule1.rs that REPLACES the planned pot/callgraph.rs internals (the phase0 module tree's callgraph.rs becomes a thin adapter). CLI path: cauldron-lint --project <root> builds a one-shot index (same collect/merge code, no service) then maps recursion_findings() into the unified Diagnostic: rule = "pot-1-recursion" (Error) / "pot-1-recursion-possible" (Advice), message carries the witness cycle as a file:line chain, citation = "JPL Power of Ten, Rule 1", fingerprint = hash(rule + ordered cycle function names) — line-independent and baseline-stable by construction. This forces the Diagnostic upgrade phase0 Area 3 already specifies (byte range + char range + line/col + fingerprint replacing today's line-only struct in lib.rs). The same snapshot feeds R9 (indirect-call + address-taken sites) so R1-Tier-2 and R9 findings share sites and waivers. IDE path: the in-app PsiService serves the identical API to the Problems panel and, later, find-usages/rename — one index, three consumers (linter CLI, linter-in-IDE, refactoring).


## Rule 1 (no recursion) check

EXACT ALGORITHM (two-tier, per the MISRA-deviation-process philosophy: sound overall, noise priced into waivers, tiers kept separate so the noisy tier never pollutes the exact one):

Nodes: linkage-keyed functions (Internal(file,name) for statics, External(name) otherwise; static/static-inline defined in headers = ONE node keyed by the header — over-merge is sound for R1), function-like macros as first-class nodes, external leaves for names with declarations but no indexed definition (libc, out-of-workspace).

Tier-1 edges (exact): every direct call_expression whose callee is a plain identifier, plus macro-mined calls. Resolution order at a call site in file F: (1) static def of that name in F -> that node, exactly; (2) else ALL external defs with that name project-wide (usually exactly 1 in partitioned cFS; if several survive partitioning, union all edges — false-positive-tolerant, adding edges is the safe direction); (3) else macro node of that name (macro-calls resolve transitively through the graph); (4) else external decl leaf. Tarjan SCC (explicit-stack implementation — the linter must not recurse) over Tier-1 edges only. Every function in a nontrivial SCC, or with a self-loop, gets ONE pot-1-recursion finding (Error, "confirmed") carrying a single witness cycle: the file:line chain of the actual call sites closing the loop (clang-tidy misc-no-recursion presentation, but cross-TU — the class NO tool in cFS CI can see: clang-tidy and cppcheck's 17.2 are both per-TU, and cFS CI runs neither with recursion checks anyway).

Tier-2 edges (sound over-approximation): every indirect call site — call_expression whose function child is not a plain identifier (deref, member access, array index) or an identifier resolving to a variable/parameter — gets edges to the ENTIRE address-taken set, filtered by ARITY when syntactically knowable (parameter counts visible at both ends; typedef'd fn-ptr types pierced via typedef stubs; the filter is DROPPED when casts/varargs obscure it, because dropping only ADDS edges — always fail toward over-approximation, never under). Address-taken = any known-function identifier in non-call position: initializer lists (dispatch tables — the cFS signature idiom), &f, assignments, and ARGUMENT position (passing a callback into OS_TaskCreate/qsort means external code may call it back — argument-position membership subsumes the external-callback problem). Re-run Tarjan over Tier-1+Tier-2; SCCs that close ONLY through indirect edges emit pot-1-recursion-possible (Advice), and each finding NAMES the specific indirect edge(s) the cycle depends on.

DECLARED ESCAPES (published verbatim in the rule doc as the soundness statement): recursion via string-symbol lookup (OS_SymbolLookup/dlsym — not even a fn-ptr expression in C; mitigated by mining cfe_es_startup.scr entry-point names into the address-taken/root set when the file is in the workspace); inline assembly; build-time-generated code not in the tree; macro-SYNTHESIZED function definitions whose bodies don't parse as C fragments (CFE_PSP_MODULE_DECLARE family). Statement: "no finding at either tier => no recursion reachable through (a) any call expression in raw source, (b) any call in an indexed macro's replacement text (one level), (c) any indirect call, ASSUMING indirect calls reach only address-taken functions. Tier 1 alone is not sound; Tier 1 + Tier 2 is sound modulo the declared escapes." That is honest, auditable, and stronger than what clang-tidy or cppcheck print about themselves.

WAIVERS (rides the Phase-4 justified-waiver system, designed now so findings carry the right identity): a waiver on the INDIRECT EDGE, not the function — // cauldron:waive rule1 "SB dispatch table, handlers verified non-reentrant, see req X" placed on the dispatch-table definition or the indirect call site — discharges EVERY Tier-2 finding whose cycle depends on that edge: one waiver kills the whole family instead of N per-function findings (the differentiator over MISRA checkers' undirected noise). Bare waiver = WAIVER-MISSING-JUSTIFICATION (Error). R9 pairing per phase0: R9 flags the same fn-ptr sites, and a justified R9 waiver on a table auto-downgrades its dependent Tier-2 R1 findings — one waiver, two rules.

FALSE-POSITIVE UX: (1) tiers visually separate — Tier-1 is near-zero-FP by construction on partitioned cFS; Tier-2 is Advice-severity and each finding shows WHY (the exact indirect edge, clickable); (2) witness cycles render as clickable file:line hop chains in the Problems panel — a human refutes a phantom cycle in seconds by walking the hops; (3) phantom-SCC prevention is structural: linkage-aware SymKey (statics can't cross-link), ut-stub partition globs (duplicate extern names can't union stub and real bodies); (4) fingerprints hash the cycle's function names, so baselines survive edits above findings; (5) refuted findings become regression fixtures via the planned cauldron-lint fp flow; (6) index-health (ERROR density, unresolved-callee counts) is surfaced next to results so "the graph is incomplete HERE" is a visible fact, not silent unsoundness.


## Editor integration

The cauldron app owns a PsiService (worker thread + Arc index + mpsc back-channel, the standard cider-PTY worker shape). Wiring:

DUMB MODE (the IntelliJ discipline, radically simplified for one process, no plugins): while status = Indexing{done,total}, every index-backed feature returns NotReady and the UI says so honestly — status bar shows "PSI: indexing 2314/5023", the Problems panel's project scope shows an "indexing..." placeholder — NEVER stale or partial whole-program answers. Tree-sitter-LOCAL features are dumb-aware by construction and stay fully live throughout: highlighting, R4 length, goto/setjmp, all single-file lint — they run on the buffer's own tree and never touch the index. The status flip is atomic (single writer), so a snapshot never sees it change mid-query. At cFS scale the dumb window is seconds, not minutes, which is exactly why the no-persistence bet is affordable.

PROBLEMS PANEL: the NASA layer (already namespaced separately from LSP per Area 3) gains a File / Project scope split. Project scope lists pot-1-recursion and pot-1-recursion-possible findings; each expands to its witness cycle as clickable file:line hops (click -> open tab at that call site); Tier-2 rows additionally show the depended-on indirect edge and, once Phase-4 waivers land, a "waive this edge" affordance. A small index-health line (N files with ERROR nodes, M unresolved callees) sits at the scope header so graph incompleteness is visible where results are read. Findings arrive from the worker via mpsc + request_repaint and are dropped if their generation is stale.

EDITORVIEW: functions in a confirmed SCC get the NASA ORANGE squiggle on the function NAME (name_range from the stub) plus the JPL gutter glyph; hover shows the cycle chain; Tier-2 renders at Advice weight. Because diagnostics ride the existing per-line diagnostic index and are mapped through the ChangeSet like LSP diagnostics, they stay anchored across edits until the next PSI pass replaces them.

EDIT FLOW END-TO-END: keystroke -> Buffer::apply marks file changed + bumps generation (nanoseconds, nothing else) -> 200 ms debounce -> worker re-collects overlay facts from the live tree -> two-hash check (body-only typing with no call change = zero further work and existing project findings stay valid) -> on real change, index update + graph recompute (ms) -> fresh findings posted -> repaint. The frame thread never blocks; the 8 ms Gate-A budget is structurally unaffected.

FUTURE (post-v1, same machinery): find-usages panel fed by the two-phase UsageStream painting progressively per file; rename = strict-resolve usages -> one multi-file Transaction batch applied end-to-start per file, with macro-body hits flagged "textual — verify" for user confirmation.

## Risks

(1) Tier-2 noise is the big unknown: cFS is ARCHITECTURALLY built on callbacks/dispatch tables, so the address-taken set may be large enough that indirect SCCs blanket the graph even arity-filtered. Mitigation is designed in: the spike measures it (with/without filter), findings group under per-table edge waivers, and the fallback — Tier-2 degrades to an "indirect-edge inventory" report while Tier-1 ships alone — still beats every tool in cFS CI. (2) The likely真 outcome that cFE/OSAL/PSP has ZERO real recursion (it is flight software) makes the demo underwhelming if framed as bug-hunting; the framing must be the acyclicity PROOF + witnessed boundary, with lab apps/tools as the likelier finding source and seeded mutations as the capability demo. (3) tree-sitter pin still unresolved (0.22 today; Gate-A ABI check may bump to 0.24/0.25): node-kind strings and the fragment-parse API could shift — keep every node-kind string in one table in collect.rs and re-run the fixture suite on any bump; psi MUST stay on the shared workspace pin. (4) ut-stub partition globs are asserted from architecture knowledge, not verified — wrong globs mean phantom SCCs (too loose) or blind spots (too tight); day-1 spike task. (5) ERROR-density on macro-heavy cFS headers could break the raw-source thesis; the 95%-clean health gate makes this a measured GO/NO-GO, not a discovered-in-production surprise, and generated build-tree headers are excluded by policy. (6) Overlay/disk divergence bugs (open-buffer facts vs watcher events racing) — contained by the single replace_file_facts chokepoint, single writer thread, and drop-overlay-on-save discipline, but needs a deliberate interleaving test. (7) Declared escapes are real unsoundness (string-symbol lookup, macro-synthesized definitions, generated code): acceptable ONLY because the soundness statement publishes them; silently widening claims later would be the real failure. (8) Solo-dev scope creep is the meta-risk: the named non-goals (no pointer analysis, no incremental SCC, no on-disk index engine, no preprocessor evaluation, no Rust v1, no stub<->AST binding) are load-bearing — each is a months-scale rabbit hole the memoized-full-recompute + in-memory design exists to avoid; revisit only with measurements in hand. (9) find-usages precision (local shadowing, statics) is good enough for navigation long before it is good enough for rename — rename ships only behind the strict phase-2 resolve with macro-hit confirmation, or it corrupts user code.
